Il Contesto

Il Problema delle Liste d'Attesa

🚫

Opacita

I cittadini non hanno modo di verificare autonomamente la propria posizione in lista ne di sapere se viene gestita correttamente.

✏️

Manipolabilita

Un database tradizionale puo essere modificato senza lasciare traccia. Spostamenti in coda, cancellazioni, retrodatazioni: tutto invisibile.

🗄️

Centralizzazione

Singolo server, singolo ente, singolo punto di fallimento. Se il server si guasta o l'ente non e trasparente, nessuno puo verificare nulla.

🤝

Fiducia Cieca

I pazienti devono fidarsi dell'ente senza strumenti di verifica indipendente.

📋

Nessun Audit Trail

Chi ha inserito quel paziente? Quando? Chi lo ha rimosso? In un DB tradizionale queste informazioni possono essere alterate.

La Nostra Risposta

Blockchain come Source of Truth

ExArt26 IOTA archivia ogni singola operazione sulla blockchain IOTA 2.0 in modo permanente e immutabile.

📦 Struttura del Payload Cifrato

Envelope V1 - Transazione Blockchain
message Dati paziente cifrati AES-256-CBC (Base64)
key Chiave AES cifrata con RSA-2048 OAEP (Base64)
iv Initialization Vector 16 byte (Base64)
hmac Codice integrita HMAC-SHA256 (Base64)
version Versione del messaggio (incrementale)

🛡️ Anche accedendo direttamente alla blockchain, si vedrebbero solo dati cifrati illeggibili senza le chiavi corrette.

Innovazione Tecnica

Codifica On-Chain u64 Split-Coin

I dati vengono archiviati interamente sulla blockchain usando una tecnica innovativa: il payload JSON cifrato viene suddiviso in chunk da 7 byte, ciascuno codificato come importo u64 in un'operazione splitCoins.

// Programmable Transaction Block
splitCoins(gas, [amount0, amount1, ... amountN])
// Struttura amounts:
amount[0] = 1           // Marker "exart26"
amount[1] = payloadLen  // Lunghezza JSON
amount[2..N] = chunk     // 1B indice + 7B dati
🧩

Dati Arbitrariamente Grandi

Supera i limiti nativi di dimensione delle transazioni IOTA suddividendo i payload in piccole unita nel formato nativo della blockchain.
